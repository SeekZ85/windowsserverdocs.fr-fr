# <a name="appendix-b-privileged-accounts-and-groups-in-active-directory"></a>Annexe b: comptes privilégiés et groupes dans ActiveDirectory

>S’applique à: Windows Server2016, Windows Server2012R2, Windows Server2012


## <a name="appendix-b-privileged-accounts-and-groups-in-active-directory"></a>Annexe b: comptes privilégiés et groupes dans ActiveDirectory  
«Privilégiés» des comptes et groupes dans Active Directory sont ceux qui autorisations et droits puissants des privilèges sont accordées qui leur permettre d’effectuer presque toute action dans Active Directory et sur les systèmes joints au domaine. Cette annexe commence par une description des privilèges, des droits et autorisations, suivi d’informations sur les comptes «le plus élevé du privilège» et les groupes dans Active Directory, autrement dit, les plus puissants et les groupes.  

Fournit également des informations sur les comptes et groupes dans Active Directory, en plus de leurs droits intégré et par défaut. Bien que les recommandations de configuration spécifiques pour sécuriser les comptes à privilèges plus élevé et les groupes sont fournies comme annexes distincts, cette annexe fournit des informations générales qui vous permet d’identifier les utilisateurs et vous devez vous concentrer sur la sécurisation des groupes. Vous procédez ainsi, car ils peuvent être exploitées par des personnes malveillantes de compromettre et même détruire votre installation d’Active Directory.  

### <a name="rights-privileges-and-permissions-in-active-directory"></a>Droits, des privilèges et autorisations dans Active Directory  
Les différences entre les droits, les autorisations et privilèges peuvent être déroutante et contradictoires, même au sein de la documentation de Microsoft. Cette section décrit quelques-unes des caractéristiques de chaque lorsqu’ils sont utilisés dans ce document. Ces descriptions pas font autoritées pour une autre documentation de Microsoft, car il peut utiliser ces termes différemment.  

#### <a name="rights-and-privileges"></a>Droits et des privilèges  
Droits et les privilèges sont en fait les mêmes fonctionnalités de l’échelle du système qui sont accordées aux principaux de sécurité tels que les utilisateurs, les services, les ordinateurs ou les groupes. Dans les interfaces généralement utilisées par les professionnels de l’informatique, ceux-ci sont généralement appelés «droits» ou «droits d’utilisateur», et ils sont souvent attribués par les objets de stratégie de groupe. La capture d’écran suivante présente certaines des droits utilisateur les plus courants qui peuvent être attribués à des entités de sécurité (il représente la stratégie par défaut des contrôleurs de domaine dans un domaine Windows Server 2012). Certaines de ces droits s’appliquent à Active Directory, tels que les **permettre aux comptes d’utilisateur et d’ordinateur à être approuvé pour délégation** droit utilisateur, tandis que les autres droits s’appliquent au système d’exploitation Windows, notamment **modifier l’heure système**.

#### <a name="permissions"></a>Autorisations  
Les autorisations sont des contrôles d’accès qui sont appliqués à des objets sécurisables tels que le système de fichiers, de Registre, services et les objets Active Directory. Chaque objet sécurisable possède une liste de contrôle d’accès associé (ACL), qui contient des entrées de contrôle d’accès (ACE) qui accordent ou refuser les principaux de sécurité (utilisateurs, services, ordinateurs ou groupes) la possibilité d’effectuer diverses opérations sur l’objet. Par exemple, les listes ACL pour de nombreux objets dans Active Directory contiennent des entrées ACE qui permettre aux utilisateurs authentifiés de lire des informations générales sur les objets, mais ne leur accordez pas la possibilité de lire les informations sensibles ou de modifier les objets.
À l’exception compte invité de chaque domaine, chaque entité de sécurité qui se connecte et est authentifiée par un contrôleur de domaine dans une forêt Active Directory ou dans une forêt approuvée a l’authentifié aux utilisateurs identificateur de sécurité (SID) ajouté à son jeton d’accès par défaut. Par conséquent, si un utilisateur, un service ou un compte d’ordinateur tente de lire les propriétés générales sur les objets utilisateur dans un domaine, l’opération de lecture est réussie.  

Si un tentatives principal de sécurité pour accéder à un objet sans les entrées sont définies et qui contiennent un SID est présent dans le jeton d’accès de l’entité de sécurité, le principal ne peut pas accéder à l’objet. En outre, si un as dans la liste ACL d’un objet contient une entrée de refus pour un SID qui correspond au jeton d’accès de l’utilisateur, ACE «Refuser» sera généralement un conflit de remplacement «autoriser» ACE. Pour plus d’informations sur le contrôle d’accès dans Windows, voir [le contrôle d’accès](https://msdn.microsoft.com/library/aa374860(v=VS.85).aspx) sur le site Web MSDN.  

Dans ce document, autorisations fait référence à des fonctionnalités qui sont accordées ou refusées aux entités de sécurité sur les objets sécurisables. Chaque fois qu’il existe un conflit entre un droit d’utilisateur et une autorisation, le droit d’utilisateur a généralement la priorité. Par exemple, si un objet dans Active Directory a été configuré avec une liste ACL qui refuse les administrateurs de tous les accès en lecture et écriture à un objet, un utilisateur qui est membre du groupe d’administrateurs du domaine ne pourra pas afficher la quantité d’informations sur l’objet. Toutefois, étant donné que le groupe Administrateurs est accordé à l’utilisateur droit «prendre possession de fichiers ou d’autres objets», l’utilisateur peut simplement prendre possession de l’objet en question, puis réécrire ACL de l’objet pour accorder un contrôle total aux administrateurs de l’objet.  

Il est pour cette raison que ce document vous recommande d’éviter à l’aide de groupes et comptes puissants pour l’administration quotidienne, plutôt que d’essayer restreindre les capacités des groupes et comptes. Il n’est pas efficacement possible d’empêcher les utilisateurs ayant accès aux informations d’identification puissantes à l’aide de ces informations d’identification pour accéder à n’importe quelle ressource sécurisable déterminés.  

### <a name="built-in-privileged-accounts-and-groups"></a>Intégré privilégié des comptes et groupes  
Active Directory est conçu pour faciliter la délégation de l’administration et le principe du privilège minimum lors de l’attribution des droits et autorisations. «Normal» aux utilisateurs qui disposent de comptes dans un domaine Active Directory sont, par défaut, en mesure de lire plus grande partie de ce qui est stocké dans le répertoire, mais sont en mesure de modifier uniquement un ensemble très limité de données dans le répertoire. Les utilisateurs qui nécessitent des privilèges supplémentaires peuvent être accordées à l’appartenance à différents groupes privilégiés qui sont intégrés dans le répertoire afin qu’ils peuvent effectuer des tâches spécifiques liées à leur rôle, mais ne peut pas effectuer des tâches qui ne sont pas pertinentes pour leurs fonctions.  

Dans Active Directory, il existe trois groupes intégrés qui composent les groupes de privilèges plus élevés dans le répertoire: le groupe Administrateurs de l’entreprise (EA), le groupe Admins du domaine (DA) et le groupe intégré Administrateurs (BA).  

Un quatrième de groupe, le groupe Administrateurs du schéma (SA), dispose de privilèges que, si l’objet d’abus, peuvent endommager ou de détruire un ensemble de la forêt Active Directory, mais ce groupe n’est plus dans ses capacités que les EA, DA et BA groupes restreint.  

Outre ces quatre groupes, il existe un nombre d’intégré supplémentaire et des groupes dans Active Directory, chacun d’eux bénéficie de droits et autorisations qui permettent à effectuer des tâches administratives spécifiques et des comptes par défaut. Bien que cette annexe ne fournit pas une présentation complète de chaque groupe intégré ou par défaut dans Active Directory, il fournit un tableau des groupes et des comptes que vous êtes probablement voir dans vos installations.  

Par exemple, si vous installez Microsoft Exchange Server dans une forêt Active Directory, groupes et comptes supplémentaires peuvent créés dans les conteneurs utilisateurs et intégré dans vos domaines. Cette annexe décrit uniquement les groupes et les comptes qui sont créés dans les conteneurs utilisateurs et intégré dans Active Directory, basé sur les fonctionnalités et rôles natifs. Comptes et groupes qui sont créés par l’installation de logiciels d’entreprise ne sont pas inclus.  

#### <a name="enterprise-admins"></a>Administrateurs de l’entreprise  
Le groupe Administrateurs de l’entreprise (EA) se trouve dans le domaine racine de forêt, et par défaut, il est membre du groupe Administrateurs intégré dans chaque domaine dans la forêt. Le compte administrateur intégré dans le domaine racine de forêt est le seul membre par défaut du groupe EA. EAs bénéficient de droits et autorisations pour les autoriser à apporter des modifications de la forêt. Il s’agit des modifications qui affectent tous les domaines de la forêt, telles que l’ajout ou suppression de domaines, établissement des approbations de forêt ou augmenter les niveaux fonctionnels de forêt. Dans un modèle de délégation correctement conçu et implémenté, l’appartenance EA est requis uniquement lors de la construction tout d’abord la forêt ou lorsque vous effectuez certaines modifications à la forêt telles que l’établissement d’une approbation de forêt sortante.  

Le groupe EA se trouve par défaut dans le conteneur utilisateurs dans le domaine racine de forêt, et il est un groupe universel de sécurité, sauf si le domaine racine de forêt est en cours d’exécution en mode mixte de Windows 2000 Server, auquel cas le groupe est un groupe de sécurité global. Bien que certains droits sont accordés directement au groupe EA, bon nombre des droits de ce groupe sont réellement héritées par le groupe EA, car il est membre du groupe Administrateurs dans chaque domaine dans la forêt. Administrateurs de l’entreprise n’ont aucuns droits par défaut sur les stations de travail ou des serveurs membres.  

#### <a name="domain-admins"></a>Admins du domaine  
Chaque domaine dans une forêt possède son propre groupe Admins du domaine (DA), qui est membre du groupe Administrateurs (BA) intégrées, de ce domaine ainsi un membre du groupe Administrateurs local sur chaque ordinateur est joint au domaine. Le seul membre par défaut du groupe DA pour un domaine est le compte administrateur intégré pour ce domaine.  

DAs sont tout puissants au sein de leurs domaines, tandis que EAs ont des privilèges de la forêt. Dans un modèle de délégation correctement conçu et implémenté, vous devez DA doit être appartenir uniquement dans les scénarios de «coupure de verre», qui sont des situations dans lesquelles un compte avec des privilèges élevés sur chaque ordinateur dans le domaine est nécessaire, ou lorsque certaines modifications large domaine doivent être effectuées. Bien que les mécanismes de délégation Active Directory natifs autorisent la délégation dans la mesure où il est possible d’utiliser des comptes DA uniquement dans les scénarios d’urgence, créant un modèle de délégation effective peut prendre beaucoup de temps et de nombreuses organisations utilisent des applications tierces pour accélérer le processus.  

Le groupe de DA est un groupe de sécurité global situé dans le conteneur utilisateurs du domaine. Il existe un seul groupe DA pour chaque domaine dans la forêt et le seul membre par défaut d’un groupe DA est le compte administrateur intégré du domaine. Étant donné que le groupe de DA d’un domaine est imbriqué dans le groupe du domaine BA et le groupe d’administrateurs local de chaque système joints au domaine, DAs ont non seulement les autorisations qui sont accordées spécifiquement aux administrateurs de domaine, mais qu’ils héritent également tous les droits et autorisations accordées au groupe d’administrateurs du domaine et le groupe Administrateurs local sur tous les systèmes joints au domaine.  

#### <a name="administrators"></a>Administrateurs  
Le groupe intégré Administrateurs (BA) est un groupe de domaine local dans le conteneur intégré d’un domaine dans lequel DAs et EAs sont imbriqués, et il est de ce groupe est accordé à la plupart des droits d’accès directs et autorisations dans le répertoire et sur les contrôleurs de domaine. Toutefois, le groupe Administrateurs pour un domaine ne dispose pas des privilèges sur les serveurs membres ou sur les stations de travail. L’appartenance au groupe Administrateurs local d’ordinateurs joints au domaine est où les privilèges locaux sont accordée; et les groupes présentés, DAs uniquement sont membres des groupes Administrateurs locaux de tous les ordinateurs joints au domaine par défaut.  

Le groupe Administrateurs est un groupe local de domaine dans le conteneur du domaine intégré. Par défaut, chaque BA groupe contient le compte d’administrateur intégré du domaine local, un groupe de DA du domaine local et groupe EA du domaine racine de forêt. Nombre de droits d’utilisateur dans Active Directory et sur les contrôleurs de domaine est accordé spécifiquement pour le groupe Administrateurs, pas à EAs ou DAs. Groupe de BA d’un domaine est accordé les autorisations Contrôle total sur la plupart des objets Active Directory et peut prendre possession des objets Active Directory. Bien que les groupes EA et DA sont accordées certaines autorisations spécifiques à un objet dans la forêt et les domaines, la puissance de groupes est effectivement «héritée» leur appartenance à des groupes BA.

> Bien que ces conditions sont les configurations par défaut de ces groupes privilégiés, un membre de l’un des trois groupes peut manipuler l’active pour adhérer à un des autres groupes. Dans certains cas, il est très facile à atteindre, tandis que dans d’autres, il est plus difficile, mais du point de vue de privilège potentiel, tous les trois groupes doivent être considérées comme équivalentes efficacement.  

#### <a name="schema-admins"></a>Administrateurs du schéma  
Le groupe Administrateurs du schéma (SA) est un groupe universel dans le domaine racine de forêt et a uniquement le compte administrateur intégré de ce domaine en tant qu’un membre par défaut, similaire au groupe EA. Bien que le groupe de l’association de sécurité permettre autoriser une personne malveillante de compromettre le schéma Active Directory, qui est l’infrastructure pour la forêt Active Directory, SAs ont peu de droits par défaut et d’autorisations au-delà du schéma.  

Vous devez soigneusement gérer et surveiller l’appartenance au groupe de l’association de sécurité, mais à certains égards, ce groupe est «moindre privilège» que les trois plus élevés privilégié groupes décrits précédemment dans la mesure où l’étendue de son privilège est très étroite; Autrement dit, SAs n’ont aucuns droits d’administration en tout lieu autre que le schéma.
