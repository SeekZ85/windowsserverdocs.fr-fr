# <a name="attractive-accounts-for-credential-theft"></a>Comptes attrayants pour le vol d'informations d'identification

>S'applique à : Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Vol de ces informations d’identification est celles dans lesquelles un attaquant parvient initialement-privilèges les plus élevés (racine, administrateur ou système, en fonction du système d’exploitation en cours d’utilisation) l’accès à un ordinateur sur un réseau, puis utilise librement les outils disponibles pour extraire des informations d’identification à partir de sessions d’autres comptes connectés. Selon la configuration du système, ces informations d’identification peuvent être extraites sous la forme de hachages, tickets ou même des mots de passe. Si des informations d’identification collectées sont pour les comptes locaux qui sont susceptibles d’exister sur d’autres ordinateurs sur le réseau (par exemple, comptes d’administrateur dans Windows ou des comptes de racine dans OSX, UNIX ou Linux), l’attaquant affiche les informations d’identification à d’autres ordinateurs sur le réseau pour propager les compromis à d’autres ordinateurs et à essayer d’obtenir les informations d’identification de deux types de comptes spécifiques :  

1.  Comptes de domaine privilégiés avec des privilèges d’étendue et approfondies (autrement dit, les comptes disposant des privilèges d’administrateur sur de nombreux ordinateurs et dans Active Directory). Ces comptes ne peuvent pas être membres d’aucun des groupes de privilège le plus élevé dans Active Directory, mais ils peuvent disposent de privilèges de niveau administrateur sur de nombreux serveurs et stations de travail dans le domaine ou la forêt, ce qui les rend efficacement aussi puissant que membres des groupes privilégiés dans Active Directory. Dans la plupart des cas, comptes qui disposent des privilèges élevés sur grandes quantités de l’infrastructure Windows étant des comptes de service, comptes de service doivent toujours être évaluées étendue et la profondeur de privilège.  

2.  « Très Important Person » comptes de domaine (VIP). Dans le contexte de ce document, un compte d’adresse IP virtuelle est un compte qui a accès aux informations d’un utilisateur malveillant veut (propriété intellectuelle et autres informations sensibles), ou n’importe quel compte peut être utilisé pour l’accès à ces informations. Voici quelques exemples de ces comptes d’utilisateur :  

    1.  Cadres dont les comptes ont accès aux informations sensibles de l’entreprise  

    2.  Comptes pour le personnel du support technique responsables de la maintenance des ordinateurs et applications utilisées par les cadres  

    3.  Comptes de service juridique qui ont accès à bid contrat les une organisation documents et si les documents sont pour leur propre organisation ou les organisations clientes  

    4.  Les chefs de produit qui ont accès aux plans et les spécifications des produits dans le développement d’une société de pipeline, quel que soit les types de produits que rend la société  

    5.  Chercheurs dont les comptes sont utilisés pour accéder aux données de l’étude, les formulations de produit ou les autres de recherche d’intérêt à une personne malveillante  

Étant donné que les comptes disposant de privilèges élevés dans Active Directory peuvent être utilisés pour propager les compromis et de manipuler des comptes d’adresse IP virtuelle ou les données qui leur sont accessibles, les comptes plus utiles pour le vol de ces informations d’identification sont des comptes qui appartiennent à des administrateurs de l’entreprise, Admins du domaine et les administrateurs des groupes dans Active Directory.  

Étant donné que les contrôleurs de domaine sont des dépôts pour la base de données AD DS et contrôleurs de domaine ont un accès complet à toutes les données dans Active Directory, les contrôleurs de domaine sont ciblés également compromis, si en parallèle avec le vol de ces informations d’identification, ou après un ou plusieurs comptes Active Directory disposant de privilèges élevés ont été compromis. Bien que plusieurs publications (et nombreux auteurs) vous concentrer sur les membres du groupe Admins du domaine lors de la description des attaques par vol de pass-the-hash et autres informations d’identification (comme décrit dans [en réduisant la Surface d’attaque Active Directory](../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md)) , un compte qui est membre d’un des groupes répertoriés ici permettre servir à compromettre l’ensemble de l’installation AD DS.

> Pour obtenir des informations complètes sur pass-the-hash et autres attaques de vol d’informations d’identification, consultez le [les attaques Mitigating Pass-the-Hash (PTH) et autres Techniques de vol d’informations d’identification](https://download.microsoft.com/download/7/7/A/77ABC5BD-8320-41AF-863C-6ECFB10CB4B9/Mitigating%20Pass-the-Hash%20(PtH)%20Attacks%20and%20Other%20Credential%20Theft%20Techniques_English.pdf) livre blanc répertorié dans [annexe M : Liens vers des documents et lecture recommandée](../../../ad-ds/manage/Appendix-M--Document-Links-and-Recommended-Reading.md). Pour plus d’informations sur les attaques par des adversaires déterminés, qui sont parfois appelé « menaces persistantes avancées » (APT), consultez [adversaires déterminés et les attaques ciblées](https://www.microsoft.com/download/details.aspx?id=34793).  

## <a name="activities-that-increase-the-likelihood-of-compromise"></a>Activités qui augmentent la probabilité de compromission  
Étant donné que la cible de vol d’informations d’identification est généralement des comptes de domaine disposant de privilèges élevés et adresse IP virtuelle, il est important pour les administrateurs à être conscients des activités qui augmentent la probabilité de réussite d’une attaque de vol d’informations d’identification. Bien que les attaquants ciblent également comptes VIP, si des adresses IP virtuelles ne sont pas fournies hauts niveaux de privilège sur les systèmes ou dans le domaine, le vol de leurs informations d’identification nécessite d’autres types d’attaques, telles que l’adresse IP virtuelle pour fournir des informations secrètes d’ingénierie sociale. Ou bien, l’attaquant doit d’abord obtenir un accès privilégié à un système sur les adresses IP virtuelles, informations d’identification sont mises en cache. Pour cette raison, les activités qui augmentent la probabilité de vol d’informations d’identification décrite ici portent principalement sur la prévention de l’acquisition d’informations d’identification d’administration disposant de privilèges élevés. Ces activités sont les mécanismes courants par laquelle les pirates informatiques sont en mesure de compromettre les systèmes pour obtenir des informations d’identification privilégiées.  

### <a name="logging-on-to-unsecured-computers-with-privileged-accounts"></a>Ouvrir une session sur des ordinateurs non sécurisés avec des comptes privilégiés  
La vulnérabilité de noyau qui permet de vol de ces informations d’identification réussir consiste à ouvrir une session sur les ordinateurs qui ne sont pas sécurisées avec les comptes qui sont largement et profondément privilégiés tout au long de l’environnement. Ces ouvertures de session peuvent être le résultat de diverses configurations inappropriées décrites ici.  

#### <a name="not-maintaining-separate-administrative-credentials"></a>Ne pas maintenir des informations d’identification d’administration distinctes  
Bien que cela soit relativement rare, à l’évaluation des différentes installations AD DS, nous avons trouvé s’ils utilisent un compte unique pour l’ensemble de leur travail. Le compte est membre d’au moins un des groupes disposant de privilèges plus élevés dans Active Directory et est le même compte que les employés utilisent pour ouvrir une session sur leurs stations de travail le matin, consulter leur messagerie, parcourir des sites Internet et télécharger contenu à leur ordinateurs. Quand les utilisateurs exécutent avec les comptes qui sont accordées les autorisations et droits d’administrateur local, ils exposent l’ordinateur local pour effectuer les compromis. Lorsque ces comptes sont membres des groupes dans Active Directory plus privilégiés, elles exposent la forêt à une compromission, rendant très simple pour un attaquant de prendre le contrôle total de l’environnement Active Directory et Windows.  

De même, dans certains environnements, nous avons constaté que les mêmes noms d’utilisateur et mots de passe sont utilisés pour les comptes de racine sur les ordinateurs non Windows sont utilisés dans l’environnement Windows, ce qui permet aux attaquants d’étendre les compromis à partir de systèmes UNIX ou Linux pour les systèmes Windows et vice versa.

#### <a name="logons-to-compromised-workstations-or-member-servers-with-privileged-accounts"></a>Ouvertures de session sur les stations de travail compromises ou des serveurs membres avec des comptes privilégiés  
Lorsqu’un compte de domaine disposant de privilèges élevés est utilisé pour la session interactive sur une station de travail compromise ou un serveur membre, cet ordinateur compromis peut-être récupérer des informations d’identification à partir de n’importe quel compte qui ouvre une session sur le système.  

#### <a name="unsecured-administrative-workstations"></a>Stations de travail d’administration non sécurisées  
Dans de nombreuses organisations, le personnel informatique utiliser plusieurs comptes. Un seul compte est utilisé pour la connexion à la station de travail de l’employé et car il s’agit de personnel informatique, ils ont souvent des droits d’administrateur local sur leurs stations de travail. Dans certains cas, l’UAC est activé afin que l’utilisateur reçoit au moins un accès partagé de gauche d’ouverture de session du jeton et devez élever lorsque des privilèges sont requises. Lors de ces utilisateurs sont les activités de maintenance, ils généralement utiliser les outils de gestion installées localement et fournir les informations d’identification pour leurs comptes de domaine avec des privilèges, en sélectionnant le **exécuter en tant qu’administrateur** option ou par en fournissant les informations d’identification lorsque vous y êtes invité. Bien que cette configuration peut sembler appropriée, il expose l’environnement à compromettre, car :  

-   Le compte d’utilisateur « standard » qui utilise de l’employé pour vous connecter à leur station de travail possède des droits d’administrateur local, l’ordinateur est vulnérable aux [de téléchargement](https://www.microsoft.com/security/sir/glossary/drive-by-download-sites.aspx) attaques dans lequel l’utilisateur est contraint à installer des logiciels malveillants.  

-   Le programme malveillant est installé dans le contexte d’un compte d’administrateur, l’ordinateur peut maintenant être utilisé pour capturer les séquences de touches, le contenu du Presse-papiers, des captures d’écran et des informations d’identification de résidant en mémoire, ce qui peut entraîner exposition les informations d’identification d’un domaine puissant compte.  

Les problèmes dans ce scénario sont doubles. Tout d’abord, bien que des comptes distincts sont utilisés pour l’administration locale et de domaine, l’ordinateur n’est pas sécurisé et ne protège pas les comptes contre le vol. En second lieu, le compte d’utilisateur normal et le compte d’administration ont reçu les autorisations et droits excessifs.  

### <a name="browsing-the-internet-with-a-highly-privileged-account"></a>Naviguer sur Internet avec un compte disposant de privilèges élevés  
Les utilisateurs qui se connectent à des ordinateurs avec des comptes qui sont membres du groupe Administrateurs local sur l’ordinateur, ou les membres des groupes privilégiés dans Active Directory, et qui puis de parcourir Internet (ou un intranet compromis) exposent l’ordinateur local et le répertoire à compromettre.  

L’accès à un site Web malveillant avec un navigateur en cours d’exécution avec des privilèges d’administrateur peut permettre à un attaquant au dépôt de code malveillant sur l’ordinateur local dans le contexte de l’utilisateur privilégié. Si l’utilisateur dispose des droits d’administrateur local sur l’ordinateur, les attaquants peuvent tromper l’utilisateur à télécharger du code nuisible ou en ouvrant les pièces jointes qui exploitent les vulnérabilités des applications et de tirer parti des privilèges de l’utilisateur pour extraire des mises en cache localement informations d’identification pour tous les utilisateurs actifs sur l’ordinateur. Si l’utilisateur dispose des droits d’administration dans le répertoire par l’appartenance à des administrateurs de l’entreprise, Admins du domaine ou des groupes d’administrateurs dans Active Directory, l’attaquant peut extraire les informations d’identification de domaine et les utiliser pour compromettre l’ensemble du domaine AD DS ou la forêt, sans avoir à compromettre un autre ordinateur de la forêt.  

### <a name="configuring-local-privileged-accounts-with-the-same-credentials-across-systems"></a>Configuration des comptes privilégiés locaux avec les mêmes informations d’identification entre les systèmes  
Configuration du même nom de compte d’administrateur local et le mot de passe sur plusieurs ou tous les ordinateurs permet aux informations d’identification volées à partir de la base de données SAM sur un ordinateur pour être utilisés pour compromettre tous les autres ordinateurs qui utilisent les mêmes informations d’identification. Au minimum, vous devez utiliser des mots de passe différents pour les comptes administrateur locaux sur chaque système joints au domaine. Comptes des administrateurs locaux peuvent également avoir des noms uniques, mais l’utilisation de mots de passe différents pour les comptes locaux privilégié de chaque système est suffisante pour vous assurer que les informations d’identification ne peuvent pas être utilisées sur d’autres systèmes.  

### <a name="overpopulation-and-overuse-of-privileged-domain-groups"></a>Overpopulation et l’utilisation abusive des groupes de domaine avec privilèges  
Accorder l’appartenance dans les groupes de EA, DA ou BA dans un domaine crée une cible pour les attaquants. Plus le nombre de membres de ces groupes, plus la probabilité qu’un utilisateur privilégié peut par inadvertance à mauvais escient les informations d’identification et les exposer le vol des informations d’identification des attaques. Chaque station de travail ou un serveur auquel se connecte un utilisateur de domaine avec privilèges présente un mécanisme possible par lequel les informations d’identification de l’utilisateur privilégié peuvent être récoltées et utilisées pour compromettre le domaine AD DS et la forêt.  

### <a name="poorly-secured-domain-controllers"></a>Contrôleurs de domaine mal sécurisé  
Contrôleurs de domaine hébergent un réplica de base de données d’un domaine AD DS. Dans le cas de contrôleurs de domaine en lecture seule, le réplica local de la base de données contient les informations d’identification pour uniquement un sous-ensemble des comptes dans le répertoire, qui sont des comptes de domaine avec privilèges par défaut. Sur les contrôleurs de domaine de lecture-écriture, chaque contrôleur de domaine maintient un réplica complet de la base de données AD DS, y compris les informations d’identification non seulement pour les utilisateurs disposant de privilèges, tels qu’Admins du domaine, mais les comptes privilégiés, tels que les comptes de contrôleur de domaine ou Krbtgt du domaine compte, qui est le compte qui est associé avec le service KDC sur les contrôleurs de domaine. Si des applications supplémentaires qui ne sont pas nécessaires pour la fonctionnalité de contrôleur de domaine sont installées sur les contrôleurs de domaine, ou si les contrôleurs de domaine ne sont pas rigoureusement corrigés et sécurisés, les attaquants peuvent compromettre les via des vulnérabilités non corrigées, ou bien ils peut tirer parti des autres vecteurs d’attaque pour installer des logiciels malveillants directement sur ces derniers.  

## <a name="privilege-elevation-and-propagation"></a>La propagation et élévation de privilèges  
Quel que soit les méthodes d’attaque utilisées, Active Directory toujours est ciblée sur des cas d’attaque d’un environnement Windows, en fin de compte contrôle l’accès à ce que les pirates souhaitable. Cela ne signifie pas que l’ensemble du répertoire est ciblé, toutefois. Des comptes spécifiques, les serveurs et les composants d’infrastructure sont généralement les cibles principales d’attaques par rapport à Active Directory. Ces comptes sont décrites comme suit.  

### <a name="permanent-privileged-accounts"></a>Comptes privilégiés permanentes  
Étant donné que l’introduction d’Active Directory, il a été possible pour des comptes d’utilisation des privilèges très élevée pour créer la forêt Active Directory, puis aux droits de délégué et des autorisations requises pour effectuer l’administration quotidienne aux comptes moins privilégiés. L’appartenance dans les groupes Administrateurs de l’entreprise, Admins du domaine ou administrateurs dans Active Directory est requis uniquement temporairement et rarement dans un environnement qui implémente les approches de moindre privilège à l’administration quotidienne.  

Les comptes privilégiés permanentes sont des comptes qui ont été placés dans des groupes privilégiés et laissés à partir d’un jour à l’autre. Si votre organisation place les cinq comptes dans le groupe Admins du domaine pour un domaine, ces cinq comptes peuvent être ciblés, 24 heures par jour, sept jours par semaine. Toutefois, la réelle nécessité d’utiliser des comptes avec des privilèges d’administrateurs du domaine est généralement uniquement pour la configuration de l’échelle du domaine spécifique et pour de courtes périodes.  

### <a name="vip-accounts"></a>Comptes d’adresse IP virtuelle  
Une cible souvent négligée en matière de violations de Active Directory est les comptes des « personnes très importants » (ou les adresses IP virtuelles) dans une organisation. Comptes privilégiés sont ciblés, car ces comptes peuvent accorder l’accès à des personnes malveillantes, ce qui leur permet de compromettre ou détruire même des systèmes ciblés, comme décrit précédemment dans cette section.  

### <a name="privilege-attached-active-directory-accounts"></a>Comptes « Attaché au privilège » Active Directory  
« Attaché au privilège » des comptes Active Directory sont des comptes de domaine qui n’ont pas été rendues affiliés à aucune des groupes qui ont les meilleurs niveaux de privilèges dans Active Directory, mais disposent à la place des niveaux élevés de privilèges sur de nombreux serveurs et stations de travail dans l’environnement. Ces comptes sont la plupart des souvent basés sur domaine qui sont configurés pour exécuter des services sur des systèmes joints à un domaine, généralement pour les applications exécutées sur grandes sections de l’infrastructure. Bien que ces comptes ont pas de privilèges dans Active Directory, si elles sont accordées doté de privilèges élevée sur un grand nombre de systèmes, elles peuvent servir à compromettre ou détruire même grands segments de l’infrastructure, atteindre le même effet que la compromission d’un compte Active Directory privilégié.
