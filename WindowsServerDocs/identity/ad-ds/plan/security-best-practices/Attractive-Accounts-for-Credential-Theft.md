# <a name="attractive-accounts-for-credential-theft"></a>Comptes intéressantes pour le vol d’informations d’identification

>S’applique à: Windows Server2016, Windows Server2012R2, Windows Server2012

Les attaques de vol d’informations d’identification sont ceux dans lesquels une personne malveillante prend initialement la plus élevée à faibles privilèges (racine, administrateur ou système, en fonction du système d’exploitation en cours d’utilisation) accès à un ordinateur sur un réseau, puis utilise disponible gratuitement des outils pour extraire des informations d’identification dans les sessions d’autres comptes connectés. En fonction de la configuration du système, ces informations d’identification peuvent être extraites sous la forme de hachages, tickets ou de mots de passe en texte en clair même. Si des informations d’identification collectées sont pour les comptes locaux qui sont susceptibles d’exister sur d’autres ordinateurs sur le réseau (par exemple, les comptes d’administrateur dans Windows ou comptes racine dans OSX, UNIX ou Linux), la personne malveillante affiche les informations d’identification à d’autres ordinateurs sur le réseau pour propager les compromis à d’autres ordinateurs et pour tenter d’obtenir les informations d’identification de deux types de comptes spécifiques:  

1.  Comptes de domaine privilégiés avec des privilèges d’étendues et approfondies (autrement dit, les comptes qui ont des privilèges d’administrateur sur de nombreux ordinateurs et dans ActiveDirectory). Ces comptes ne peuvent pas être membres de n’importe lequel des groupes de privilège le plus élevé dans ActiveDirectory, mais ils peuvent disposent de privilèges de niveau administrateur sur de nombreux serveurs et stations de travail dans le domaine ou la forêt, ce qui les rend efficacement aussi puissant que les membres des groupes privilégiés dans ActiveDirectory. Dans la plupart des cas, les comptes qui ont reçu des privilèges élevés sur les grandes quantités de l’infrastructure Windows sont les comptes de service, les comptes de service doivent toujours être évaluées étendue et la profondeur de privilège.  

2.  «Personne très importante» des comptes de domaine (VIP). Dans le contexte de ce document, un compte d’adresse IP virtuelle est un compte qui a accès aux informations d’une personne malveillante souhaite (droits de propriété intellectuelle et d’autres informations sensibles), ou n’importe quel compte peut être utilisé pour accorder l’accès aux informations. Voici quelques exemples de ces comptes d’utilisateur:  

    1.  Dirigeants dont les comptes ont accès aux informations sensibles de l’entreprise  

    2.  Comptes pour le personnel du support technique qui est chargé de gérer les ordinateurs et les applications utilisées par des dirigeants  

    3.  Comptes de service juridique qui ont accès aux documents d’enchère et contrat d’une organisation, que les documents pour leur propre organisation ou les organisations de client  

    4.  Planificateurs de produit qui ont accès à des plans et les spécifications de produits dans le développement d’une société de pipeline, quel que soit les types de produits que rend la société  

    5.  Fonctions de recherche dont les comptes sont utilisés pour accéder aux données étude, produits ou autres de recherche de vos centres d’intérêt à une personne malveillante  

Étant donné que les comptes dotés de privilèges élevés dans ActiveDirectory peuvent être utilisés de propager la compromission et de manipuler les comptes d’adresse IP virtuelle ou les données qu’ils peuvent accéder, les comptes plus utiles pour les attaques de vol d’informations d’identification sont des comptes qui sont membres des groupes Administrateurs, Admins du domaine et administrateurs de l’entreprise dans ActiveDirectory.  

Étant donné que les contrôleurs de domaine sont les référentiels pour la base de données ADDS et les contrôleurs de domaine ont un accès complet à toutes les données dans ActiveDirectory, les contrôleurs de domaine sont également ciblés en cas de compromission, si en parallèle avec les attaques de vol d’informations d’identification, ou après un ou plusieurs privilèges élevés ActiveDirectory comptes ont été compromis. Bien que plusieurs publications (et nombreuses personnes malveillantes) se concentrent sur les appartenances aux groupes Admins du domaine lorsque vous décrivez les attaques par vol de pass-the-hash et autres informations d’identification (comme décrit dans [ce qui réduit la Surface d’attaque ActiveDirectory](../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md)), un compte qui est membre d’un des groupes répertoriés ici permettre servir à compromettre l’intégralité de l’installation ADDS.

## <a name="activities-that-increase-the-likelihood-of-compromise"></a>Activités qui augmentent le risque de compromission  
Étant donné que la cible de vol d’informations d’identification est généralement des comptes de domaine dotés de privilèges élevés et adresses IP virtuelles, il est important pour les administrateurs à être conscient des activités qui augmentent la probabilité de réussite d’une attaque par usurpation d’informations d’identification. Bien que les personnes malveillantes également ciblent des comptes d’adresse IP virtuelle, si les adresses IP ne figurent pas des niveaux élevés de privilèges sur des systèmes ou dans le domaine, le vol de leurs informations d’identification nécessite d’autres types d’attaques, telles que l’adresse IP virtuelle pour fournir des informations secrètes d’ingénierie ingénierie. Ou bien la personne malveillante doit d’abord obtenir un accès privilégié à un système sur les adresses IP virtuelles les informations d’identification sont mises en cache. Pour cette raison, les activités qui augmentent le risque de vol d’informations d’identification décrit ici visent principalement sur la prévention de l’acquisition des informations d’identification d’administration dotés de privilèges élevés. Ces activités sont les mécanismes courants par lequel les personnes malveillantes sont en mesure de compromettre les systèmes pour obtenir des informations d’identification privilégiées.  

### <a name="logging-on-to-unsecured-computers-with-privileged-accounts"></a>Ouverture de session sur des ordinateurs non sécurisés avec des comptes privilégiés  
La vulnérabilité de base qui permet les attaques de vol d’informations d’identification réussir consiste à ouvrir une session sur les ordinateurs qui ne sont pas sécurisés avec des comptes qui sont largement et profondément privilégié tout au long de l’environnement. Ces ouvertures de session peuvent être le résultat de diverses erreurs de configuration décrits ici.  

#### <a name="not-maintaining-separate-administrative-credentials"></a>Maintenance ne pas les informations d’identification d’administration distinctes  
Bien que cela soit relativement rare, pour l’évaluation des différentes installations ADDS, nous avons constaté employés informatique à l’aide d’un seul compte pour l’ensemble de leur travail. Le compte est membre d’au moins un des groupes dotés de privilèges plus élevés dans ActiveDirectory est le même compte que l’utilisation des employés à ouvrir une session sur leurs stations de travail du matin, vérifiez son adresse e-mail, parcourir des sites Internet et télécharger le contenu à leurs ordinateurs. Lorsque les utilisateurs exécutent avec des comptes disposant des autorisations et droits d’administrateur local, ils exposent l’ordinateur local pour effectuer des compromis. Lorsque ces comptes sont également membres des groupes privilégiés plus dans ActiveDirectory, ils exposent la forêt à une compromission, ce qui facilite le plus simplement pour un attaquant d’acquérir le contrôle complet de l’environnement ActiveDirectory et de Windows.  

De même, dans certains environnements, nous avons réalisé que les mêmes noms d’utilisateur et mots de passe sont utilisés pour les comptes de racine sur les ordinateurs non Windows sont utilisés dans l’environnement Windows, ce qui permet aux attaquants d’étendre de compromission des systèmes UNIX ou Linux pour les systèmes Windows et vice versa.

#### <a name="logons-to-compromised-workstations-or-member-servers-with-privileged-accounts"></a>Ouvertures de session sur les stations de travail compromises ou les serveurs membres avec des comptes privilégiés  
Lorsqu’un compte de domaine disposant de privilèges élevés est utilisé pour la session interactive sur une station de travail compromise ou un serveur membre, cet ordinateur compromis peut collecter des informations d’identification à partir de n’importe quel compte qui ouvre une session sur le système.  

#### <a name="unsecured-administrative-workstations"></a>Stations de travail d’administration non sécurisées  
Dans de nombreuses organisations, le personnel informatique utiliser plusieurs comptes. Un compte est utilisé pour la connexion à la station de travail de l’employé, et dans la mesure où il s’agit personnel informatique, ils ont souvent des droits d’administrateur local sur leurs stations de travail. Dans certains cas, l’UAC est activé afin que l’utilisateur reçoit au moins un accès fractionné de gauche d’ouverture de session de jeton et doit élever lorsque des privilèges sont requises. Lors de ces utilisateurs sont des activités de maintenance, ils généralement utiliser des outils de gestion installés localement et fournir les informations d’identification pour leurs comptes de domaine doté de privilèges, en sélectionnant le **exécuter en tant qu’administrateur** option ou en fournissant les informations d’identification lorsque vous y êtes invité. Bien que cette configuration peut sembler appropriée, il expose l’environnement à compromettre car:  

-   Le compte d’utilisateur «normal» par l’employé pour vous connecter à sa station de travail dispose des droits d’administrateur local, l’ordinateur est vulnérable aux [de téléchargement](https://www.microsoft.com/security/sir/glossary/drive-by-download-sites.aspx) attaques dans lequel l’utilisateur est contraint à installer des logiciels malveillants.  

-   Le logiciel malveillant est installé dans le contexte d’un compte administratif, l’ordinateur peut maintenant être utilisé pour capturer les frappes de touches, le contenu du Presse-papiers, captures d’écran et informations d’identification résident, qui peut entraîner l’exposition des informations d’identification d’un compte de domaine puissant.  

Les problèmes dans ce scénario sont double. Tout d’abord, bien que des comptes distincts sont utilisés pour l’administration locale et de domaine, l’ordinateur est sécurisé et ne protège pas les comptes contre le vol. Ensuite, le compte d’utilisateur standard et le compte d’administration ont reçu les autorisations et droits excessifs.  

### <a name="browsing-the-internet-with-a-highly-privileged-account"></a>La navigation sur Internet avec un compte à privilèges élevés  
L’ordinateur local et le répertoire de compromettre exposent les utilisateurs pour ouvrir une session sur les ordinateurs avec des comptes qui sont membres du groupe Administrateurs local sur l’ordinateur, ou les membres de groupes privilégiés dans ActiveDirectory et qui puis parcourir Internet (ou un intranet compromis).  

L’accès à un site Web malveillant avec un navigateur en cours d’exécution avec des privilèges d’administration peut permettre à un attaquant verser des programmes malveillants sur l’ordinateur local dans le contexte de l’utilisateur privilégié. Si l’utilisateur dispose des droits d’administrateur local sur l’ordinateur, les pirates peuvent inciter l’utilisateur à télécharger du code malveillant ou ouvrir des pièces jointes que tirer parti des vulnérabilités des applications et de tirer parti des privilèges de l’utilisateur pour extraire localement mis en cache les informations d’identification pour tous les utilisateurs actifs sur l’ordinateur. Si l’utilisateur dispose de droits d’administration dans le répertoire par l’appartenance dans les administrateurs de l’entreprise, Admins du domaine ou les groupes d’administrateurs dans ActiveDirectory, la personne malveillante peut extraire les informations d’identification de domaine et les utiliser pour compromettre l’intégralité du domaine ADDS ou la forêt, sans avoir à compromettre un autre ordinateur de la forêt.  

### <a name="configuring-local-privileged-accounts-with-the-same-credentials-across-systems"></a>Configurer les comptes privilégiés Local avec les mêmes informations d’identification sur les systèmes  
La configuration du même nom du compte administrateur local et le mot de passe sur plusieurs ou tous les ordinateurs permet aux informations d’identification volées à partir de la base de données SAM sur un ordinateur pour être utilisés pour compromettre tous les autres ordinateurs qui utilisent les mêmes informations d’identification. Au minimum, vous devez utiliser des mots de passe différents pour les comptes administrateur locaux sur chaque système joints au domaine. Comptes d’administrateur local peuvent également porter un nom unique, mais à l’aide de mots de passe différents pour les comptes locaux privilégié de chaque système est suffisant pour vous assurer que les informations d’identification ne peuvent pas être utilisées sur d’autres systèmes.  

### <a name="overpopulation-and-overuse-of-privileged-domain-groups"></a>Overpopulation et une utilisation abusive des groupes de domaine privilégiés  
L’octroi d’appartenance dans les groupes EA, DA ou BA dans un domaine crée une cible pour les pirates. Plus le nombre de membres de ces groupes, Pluss la probabilité qu’un utilisateur privilégié peut par inadvertance les informations d’identification à mauvais escient et les exposer vol des informations d’identification des attaques. Chaque station de travail ou un serveur auquel se connecte un utilisateur de domaine privilégiés présente un mécanisme possible qui informations d’identification de l’utilisateur privilégié peuvent être récoltées et compromettre le domaine ADDS et la forêt.  

### <a name="poorly-secured-domain-controllers"></a>Contrôleurs de domaine mal sécurisés  
Contrôleurs de domaine hébergent un réplica de base de données des services ADDS d’un domaine. Dans le cas des contrôleurs de domaine en lecture seule, le réplica de la base de données local contient les informations d’identification uniquement un sous-ensemble des comptes dans le répertoire, qui sont des comptes de domaine privilégiés par défaut. Sur les contrôleurs de domaine en lecture-écriture, chaque contrôleur de domaine gère un réplica complet de la base de données ADDS, y compris les informations d’identification non seulement pour les utilisateurs privilégiés comme Admins du domaine, mais les comptes privilégiés, tels que les comptes de contrôleur de domaine ou compte Krbtgt du domaine, qui est le compte qui est associé le KDC de service sur les contrôleurs de domaine. Si des applications supplémentaires qui ne sont pas nécessaires pour la fonctionnalité du contrôleur de domaine sont installées sur les contrôleurs de domaine, les contrôleurs de domaine ne sont pas strictement corrigées et sécurisés, des attaquants peuvent compromettre les vulnérabilités non corrigées par le biais, ou ils peuvent tirer parti des autres vecteurs d’attaque pour installer des logiciels malveillants directement sur.  

## <a name="privilege-elevation-and-propagation"></a>La propagation et élévation de privilèges  
Quel que soit les méthodes d’attaque utilisées, ActiveDirectory est toujours destiné lorsqu’un environnement Windows est attaqué, car il contrôle en fin de compte d’accès à ce que les pirates choix. Cela ne signifie pas que l’intégralité du répertoire est ciblé, toutefois. Des comptes spécifiques, les serveurs et les composants d’infrastructure sont généralement les cibles principales d’attaques par rapport à ActiveDirectory. Ces comptes sont décrites ci-dessous.  

### <a name="permanent-privileged-accounts"></a>Comptes privilégiés permanentes  
Étant donné que l’introduction d’ActiveDirectory, il a été possible aux comptes d’utilisation des privilèges très élevée pour créer la forêt ActiveDirectory, puis au délégué droits et autorisations nécessaires pour effectuer l’administration quotidienne aux comptes de moins de privilèges. L’appartenance dans les groupes Administrateurs, Admins du domaine ou administrateurs de l’entreprise dans ActiveDirectory est requis uniquement temporairement et rarement dans un environnement qui implémente les approches de privilèges à l’administration quotidienne.  

Comptes privilégiés permanents sont des comptes qui ont été placés dans des groupes privilégiés et laissés au quotidien. Si votre organisation place les cinq comptes dans le groupe Admins du domaine pour un domaine, les cinq comptes peuvent être ciblés, 24heures sur 24, sept jours par semaine. Toutefois, le réel besoin d’utiliser des comptes avec des privilèges d’administrateurs du domaine est généralement uniquement pour une configuration spécifique à l’échelle du domaine et pour de courtes périodes.  

### <a name="vip-accounts"></a>Comptes VIP  
Une cible de violations d’ActiveDirectory souvent négligée est les comptes de «personnes très importants» (ou VIP) dans une organisation. Comptes privilégiés sont ciblés dans la mesure où ces comptes peuvent accorder l’accès à des personnes malveillantes, ce qui leur permet de compromettre ou détruire même systèmes ciblés, comme décrit précédemment dans cette section.  

### <a name="privilege-attached-active-directory-accounts"></a>«Attaché au privilège» les comptes ActiveDirectory  
«Attaché au privilège» les comptes ActiveDirectory sont des comptes de domaine qui n’ont pas été apportées membres de n’importe lequel des groupes qui ont des niveaux de privilèges dans ActiveDirectory, mais disposent à la place des niveaux élevés de privilèges sur plusieurs serveurs et stations de travail dans l’environnement. Ces comptes sont plus souvent basés sur un domaine qui sont configurés pour exécuter les services sur des systèmes joints au domaine, généralement pour les applications qui s’exécutent sur des sections importantes de l’infrastructure. Bien que ces comptes n’ont aucuns privilèges dans ActiveDirectory, si elles sont accordées à privilèges élevés sur un grand nombre de systèmes, ils peuvent être utilisés pour compromettre ou détruire même des segments de grande taille de l’infrastructure, atteindre le même effet que la compromission d’un compte privilégié d’ActiveDirectory.
