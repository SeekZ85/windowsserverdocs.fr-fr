# <a name="planning-for-compromise"></a>Planification des compromis

>S'applique à : Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

*La loi numéro un : Personne ne pense que quoi que ce soit incorrecte peut se produire, jusqu'à ce que c’est le cas.* - [10 lois immuables de l’Administration de la sécurité](https://technet.microsoft.com/library/cc722488.aspx)  
  
Les plans de récupération d’urgence dans de nombreuses organisations se concentrer sur la récupération des sinistres régionaux défaillances d’ou entraîner une perte de services informatiques de. Toutefois, lorsque vous travaillez avec les clients compromis, il est fréquent que la récupération à partir d’une compromission intentionnelle est absente dans leurs plans de récupération d’urgence. Cela est particulièrement vrai lorsque la compromission entraîne le vol de propriété intellectuelle ou une destruction intentionnelle qui s’appuie sur les limites logiques (par exemple, la destruction de tous les domaines Active Directory ou de tous les serveurs) plutôt que des limites physiques (tel que destruction d’un centre de données). Bien qu’une organisation peut avoir des plans de réponse aux incidents qui définissent les activités initiales à entreprendre lorsqu’un compromis est découvert, ces plans omettre souvent les étapes pour récupérer d’un compromis qui affecte l’ensemble de l’infrastructure informatique.  
  
Étant donné que Active Directory fournit les riches fonctionnalités de gestion des identités et des accès pour les utilisateurs, des serveurs, des stations de travail et des applications, il est invariablement cible d’attaquants. Si une personne malveillante obtient un accès hautement privilégié à un domaine Active Directory ou un contrôleur de domaine, que l’accès peut être exploité pour y accéder, contrôler ou même détruire l’ensemble de la forêt Active Directory.  
  
Ce document a abordé certaines des attaques courantes contre Windows et Active Directory et des contre-mesures que vous pouvez implémenter pour réduire votre surface d’attaque, mais la seule façon pour récupérer en cas de compromission complète d’Active Directory doit être préparé pour la compromission avant qu’elle se produise. Cette section concentre sur les détails d’implémentation technique que les sections précédentes de ce document, et plus sur les recommandations de haut niveau que vous pouvez utiliser pour créer une approche holistique et complète pour sécuriser et gérer votre organisation de critiques entreprise et les ressources informatiques.  
  
Si votre infrastructure n’a jamais été attaquée, a résista a tenté de violations, ou succombé aux attaques et été entièrement compromis, vous devez planifier la réalité inévitable que vous être attaqué encore et encore. Il n’est pas possible d’empêcher les attaques, mais il peut en effet être possible d’empêcher les violations significatives ou compromission en gros. Chaque organisation doit étroitement évaluer leurs programmes de gestion des risques existants et effectuez les ajustements nécessaires afin de réduire leur niveau global de vulnérabilité en apportant des investissements à charge équilibrée dans prévention, de détection, de relation contenant-contenu et de récupération.  
  
Pour créer une défense efficace tout en fournissant des services aux utilisateurs et les entreprises qui dépendent de votre infrastructure et vos applications, vous pouvez besoin à prendre en compte les nouvelles méthodes pour empêcher, détecter et contiennent des compromis dans votre environnement et ensuite récupérer à partir la compromission. Les approches et les recommandations dans ce document ne peuvent pas vous aider à réparer une installation de Active Directory compromise, mais peuvent vous aider à sécuriser votre application suivante.  
  
Recommandations pour la récupération d’une forêt Active Directory sont présentées dans [Windows Server 2012 : Planification de la récupération de forêt Active Directory](https://www.microsoft.com/download/details.aspx?id=16506). Vous pourrez peut-être empêcher votre nouvel environnement à partir de la compromission complètement, mais même si vous ne pouvez pas le cas, vous devez les outils nécessaires pour récupérer et de reprendre le contrôle de votre environnement.  
  
## <a name="rethinking-the-approach"></a>Repenser l’approche  
*La loi numéro huit : La difficulté de défense d’un réseau est directement proportionnelle à sa complexité.* - [10 lois immuables de l’Administration de la sécurité](https://technet.microsoft.com/library/cc722488.aspx)  
  
Il est généralement bien acceptée que si une personne malveillante a obtenu système, administrateur, racine ou équivalent à un ordinateur, quel que soit le système d’exploitation, cet ordinateur peut ne plus être considéré comme digne de confiance, quel que soit le nombre d’efforts sont effectuées pour « nettoyer » le système. Active Directory n’est pas différent. Si une personne malveillante a obtenu un accès privilégié à un contrôleur de domaine ou un compte à privilèges élevés dans Active Directory, sauf si vous avez un enregistrement de chaque modification, il procède à ou d’une sauvegarde reconnue fiable, vous ne pouvez jamais restaurer le répertoire à un complètement état de confiance.  
  
Lorsqu’un serveur membre ou une station de travail est compromise et modifiée par une personne malveillante, l’ordinateur n’est plus fiable, mais les stations de travail et serveurs sans compromis voisines peuvent toujours être approuvées. Corruption d’un ordinateur n’implique pas que tous les ordinateurs sont compromis.  
  
Toutefois, dans un domaine Active Directory, tous les contrôleurs de domaine hébergent des réplicas de la même base de données AD DS. Si un seul contrôleur de domaine est compromis et qu’une personne malveillante modifie la base de données AD DS, ces modifications se répliquer sur chaque contrôleur de domaine dans le domaine et en fonction de la partition dans laquelle les modifications sont apportées, la forêt. Même si vous réinstallez chaque contrôleur de domaine dans la forêt, vous réinstallez simplement les ordinateurs hôtes sur lequel réside la base de données AD DS. Les modifications malveillantes de Active Directory répliquera aux contrôleurs de domaine nouvellement installée aussi facilement que si elles seront répliqués sur les contrôleurs de domaine qui ont été exécutées pendant des années.  
  
Évaluer les environnements compromis, nous couramment trouvé que ce qui a été considérée comme la première violation « événement » a été réellement déclenchée après semaines, mois, voire même des années après que les attaquants avaient compromis initialement l’environnement. Les attaquants généralement obtenu les informations d’identification pour les comptes à privilèges élevés long avant une violation a été détectée, et ils exploité ces comptes pour compromettre l’annuaire, les contrôleurs de domaine, les serveurs membres, les stations de travail et même connectés non Windows systèmes.  
  
Ces résultats sont cohérents avec plusieurs conclusions dans 2012 violation enquêtes rapport de Verizon de données, qui stipule que :  
  
-   98 % des violations de données se définit comme des agents externes  
  
-   85 % des violations de données a duré semaines ou autres services  
  
-   92 pour cent des incidents ont été découvertes par un tiers, et  
  
-   97 % des violations ont été évitable si simple ou intermédiaire des contrôles.  
  
Un compromis au degré décrit précédemment est irréparable efficacement et le Conseil standard de « remanier et reconfigurer » tous les systèmes compromis sont tout simplement pas possible ou même possible que si Active Directory n’a pas été compromis ou détruit. Même restauration à un état correct connu n’élimine pas les failles qui autorisées de l’environnement à être compromis en premier lieu.  
  
Bien que vous devez protéger toutes les facettes de votre infrastructure, un attaquant doit uniquement trouver suffisamment défauts dans vos défenses pour atteindre leur objectif souhaité. Si votre environnement est relativement simple et trouvés et historiquement bien gérée, puis en implémentant les recommandations fournies précédemment dans ce document peut être une proposition simple.  
  
Toutefois, nous avons constaté que les plus anciennes, plus grande et plus complexes l’environnement, plus il est probable que les recommandations de ce document sera irréalisable ou même impossible à implémenter. Il est beaucoup plus difficile à sécuriser une infrastructure après le fait qu’il est à mettre à jour et de construire un environnement qui résiste aux attaques et compromettre. Mais comme indiqué précédemment, aucune entreprise petite reconstruire l’intégralité d’une forêt Active Directory. Pour ces raisons, nous recommandons plus le focus, l’approche ciblée pour sécuriser vos forêts Active Directory.  
  
Au lieu de se concentrer sur et essayer de résoudre tous les aspects qui sont « interrompue », envisagez une approche dans laquelle vous donnez la priorité selon ce qui est plus important pour votre entreprise et de votre infrastructure. Au lieu de tenter de corriger un environnement rempli avec les applications et systèmes obsolètes, mal configurés, envisagez de créer un nouvel environnement de petite et sécurisé dans lequel vous pouvez déplacer en toute sécurité les utilisateurs, les systèmes, ainsi que les informations les plus importants de votre entreprise.  
  
Dans cette section, nous décrivons une approche selon laquelle vous pouvez créer une forêt AD DS initial qui sert en tant que « bateau vie » ou « cellule sécurisée » pour votre infrastructure d’entreprise core. Une forêt vierge est simplement une nouvellement installée forêt d’Active Directory qui est généralement limité en taille et la portée et qui est intégré à l’aide des systèmes d’exploitation actuels, les applications et avec les principes décrits dans [réduction actif Surface d’attaque Directory](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md).  
  
En implémentant les paramètres de configuration recommandées dans une forêt nouvellement générée, vous pouvez créer une installation AD DS qui est conçue à partir d’a à z avec les paramètres de sécurité et pratiques, et vous pouvez réduire les difficultés qui accompagnent la prise en charge des systèmes hérités et applications. Alors que des instructions détaillées pour la conception et l’implémentation d’une installation AD DS trouvée sont en dehors de la portée de ce document, vous devez suivre certains principes généraux et des instructions permettant de créer une « cellule sécurisée » dans lequel vous pouvez héberger votre plus critiques ressources. Ces directives sont les suivantes :  
  
1.  Identifier les principes de séparation et de sécurisation des ressources critiques.  
  
2.  Définir un plan de migration limitées, en fonction des risques.  
  
3.  Tirez parti des migrations « nonmigratory » lorsque cela est nécessaire.  
  
4.  Implémenter des « destruction creative ».  
  
5.  Isoler les applications et les systèmes hérités.  
  
6.  Simplifiez la sécurité pour les utilisateurs finaux.  
  
### <a name="identifying-principles-for-segregating-and-securing-critical-assets"></a>Identification des principes de séparation et de sécurisation des ressources critiques  

Les caractéristiques de l’environnement initial que vous créez à des ressources critiques maison peuvent varier considérablement. Par exemple, vous pouvez choisir de créer une forêt vierge dans laquelle vous migrez uniquement les utilisateurs de l’adresse IP virtuelle et les données sensibles que seuls les utilisateurs peuvent accéder. Vous pouvez créer une forêt vierge dans laquelle vous ne migrez pas uniquement les utilisateurs d’adresse IP virtuelle, mais que vous implémentez dans une forêt d’administration, mise en œuvre les principes décrits dans [en réduisant la Surface d’attaque Active Directory](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md) créer sécurisé comptes d’administration et les hôtes qui peuvent être utilisées pour gérer vos forêts hérités à partir de la forêt d’origine. Vous pouvez implémenter une forêt « spécialisée » qui héberge les comptes, les comptes privilégiés et les systèmes nécessitant une sécurité supplémentaire, tels que les serveurs exécutant les Services de certificats Active Directory (AD CS) dans le but unique de séparant ainsi ces derniers moins sécurisé VIP forêts. Enfin, vous pouvez implémenter une forêt vierge qui devient l’emplacement de fait pour tous les nouveaux utilisateurs, systèmes, applications et données, ce qui vous permet de finalement désaffecter votre forêt héritée via l’usure.  
  
Indépendamment de si votre forêt initial contient un certain nombre d’utilisateurs et systèmes ou il constitue la base pour une migration plus agressive, vous devez suivre ces principes dans votre planification :  
  
1.  Partons du principe que vos forêts hérités ont été compromis.  
  
2.  Ne configurez pas un environnement initial pour approuver une forêt héritée, bien que vous pouvez configurer un environnement hérité pour approuver une forêt vierge.  
  
3.  Ne migrent pas des comptes d’utilisateur ou des groupes à partir d’une forêt héritée dans un environnement initial s’il existe un risque que les appartenances du compte, l’historique des SID ou autres attributs ont peut-être été à des fins malveillantes modifiées. Au lieu de cela, utilisez « nonmigratory » approches pour remplir une forêt vierge. (Nonmigratory approches sont décrites plus loin dans cette section.)  
  
4.  Ne migrent pas les ordinateurs à partir de forêts hérités aux forêts trouvés. Implémenter des serveurs nouvellement installées dans la forêt vierge, installer des applications sur les serveurs nouvellement installées et migrer des données d’application pour les systèmes nouvellement installés. Pour les serveurs de fichiers, copier des données vers des serveurs nouvellement installées, définir des ACL à l’aide d’utilisateurs et groupes dans la nouvelle forêt, puis créer des serveurs d’impression de la même manière.  
  
5.  N’autorisent pas l’installation de systèmes d’exploitation hérités ou applications de la forêt d’origine. Si une application ne peut pas être mis à jour et qui vient d’être installée, laisser dans la forêt héritée et destruction creative pour remplacer la fonctionnalité d’application.  
  
### <a name="defining-a-limited-risk-based-migration-plan"></a>Définition d’un Plan de Migration limitées, en fonction des risques  
Création d’une limitée, plan de migration en fonction des risques signifie simplement que lorsque vous décidez quels utilisateurs, applications et des données à migrer dans votre forêt vierge, vous devez identifier les cibles de migration basées sur le degré de risque à laquelle votre organisation est exposée si un des les utilisateurs ou les systèmes est compromis. Utilisateurs VIP dont les comptes sont plus susceptibles d’être la cible d’attaquants doivent résider dans la forêt d’origine. Les applications qui fournissent des fonctions métier essentielles doivent être installées sur des serveurs qui vient d’être développés dans la forêt d’origine, et des données très sensibles doivent être déplacées vers les serveurs dans la forêt vierge sécurisé.  
  
Si vous ne disposez pas d’une vision claire des utilisateurs plus stratégiques, systèmes, applications et données dans votre environnement Active Directory, fonctionnent avec les divisions pour identifier les. N’importe quelle application requise pour l’entreprise pour fonctionner doit être identifiée, de même que tous les serveurs sur lesquels exécutent des applications critiques ou des données critiques sont stockées. En identifiant les utilisateurs et les ressources qui sont requises pour votre organisation continuer à fonctionner, vous créer une collection hiérarchisée naturellement de ressources sur lequel se concentrer vos efforts.  
  
### <a name="leveraging-nonmigratory-migrations"></a>En tirant parti des Migrations « Nonmigratory »  
Si vous savez que votre environnement a été compromis, pensez qu’il a été compromise, ou simplement préférez ne pas migrer les anciennes données et les objets à partir d’une installation d’Active Directory héritée vers un autre, prenez en compte les approches de migration qui n’est pas techniquement objets « migrer ».  
  
### <a name="user-accounts"></a>Comptes d'utilisateurs  
Dans une migration traditionnelle d’Active Directory à partir d’une forêt vers un autre, l’attribut SIDHistory (historique SID) sur des objets utilisateur est utilisé pour stocker les SID des utilisateurs et les SID des groupes que les utilisateurs étaient des membres dans la forêt héritée. Si les comptes d’utilisateurs sont migrés vers une nouvelle forêt, et ils accèdent aux ressources dans la forêt héritée, les SID de l’historique SID sont utilisés pour créer un jeton d’accès qui permet aux utilisateurs d’accéder aux ressources auxquelles ils avaient accès avant que les comptes ont été migrés.  
  
Gestion de l’historique des SID, toutefois, se sont révélé une problématique dans certains environnements, car le remplissage des jetons d’accès utilisateurs avec des SID actuelles et historiques peut entraîner un encombrement jeton. Encombrement de jeton est un problème dans lequel le nombre d’identificateurs de sécurité qui doivent être stockées dans un jeton d’accès utilisateur utilise ou dépasse la quantité d’espace disponible dans le jeton.  
  
Bien que la taille des jetons peut être augmentée à une certaine mesure, la solution parfaite à encombrement jeton consiste à réduire le nombre de SID associés aux comptes d’utilisateur, que ce soit par la rationalisation des appartenances aux groupes, éliminant l’historique des SID, ou une combinaison des deux. Pour plus d’informations sur l’encombrement de jeton, consultez [MaxTokenSize et l’encombrement du jeton Kerberos](http://blogs.technet.com/b/shanecothran/archive/2010/07/16/maxtokensize-and-kerberos-token-bloat.aspx).  
  
Plutôt que de la migration des utilisateurs à partir d’un environnement hérité (en particulier un dans les appartenances aux groupes et les SID des historiques peuvent être compromis) à l’aide de l’historique des SID, envisagez l’utilisation d’applications de méta-annuaire à « migrer » les utilisateurs, sans avoir à insérer des historiques de SID dans la nouvelle forêt. Lorsque des comptes d’utilisateur sont créées dans la nouvelle forêt, vous pouvez utiliser une application de méta-annuaire pour mapper les comptes à leurs comptes correspondants dans la forêt héritée.  
  
Pour fournir le nouvel utilisateur comptes l’accès aux ressources dans la forêt héritée, vous pouvez utiliser les outils de méta-annuaire pour identifier les groupes de ressources dans lequel les anciens comptes d’utilisateurs voyait accorder l’accès et puis ajoutez les nouveaux comptes d’utilisateurs à ces groupes. En fonction de votre stratégie de groupe dans la forêt héritée, vous devrez peut-être créer des groupes locaux de domaine pour accéder aux ressources ou de convertir les groupes existants aux groupes locaux de domaine pour autoriser les nouveaux comptes doit être ajouté aux groupes de ressources. En se concentrant tout d’abord sur les applications les plus critiques et les données et leur migration vers le nouvel environnement (avec ou sans l’historique des SID), vous pouvez limiter la quantité d’effort consenti dans l’environnement hérité.  
  

  
### <a name="servers-and-workstations"></a>Serveurs et stations de travail  
Dans une migration traditionnelle d’un annuaire Active Directory forêt sur des ordinateurs d’un autre, la migration est souvent relativement simple par rapport à la migration des utilisateurs, groupes et applications. En fonction du rôle d’ordinateur, la migration vers une nouvelle forêt peut être aussi simple que séparé, un ancien domaine et accéder à une nouvelle. Toutefois, la migration intactes les comptes d’ordinateur dans un defeats forêt trouvés le but de créer un nouvel environnement. Plutôt que de comptes d’ordinateurs de (potentiellement compromis, mal configurés ou obsolètes) migration vers une nouvelle forêt, vous devez installer fraîchement serveurs et stations de travail dans le nouvel environnement. Vous pouvez migrer des données à partir de systèmes dans la forêt héritée pour les systèmes dans la forêt d’origine, mais pas les systèmes hébergeant les données.  
  
### <a name="applications"></a>Applications  

Applications peuvent présenter le défi plus significatif dans toute migration à partir d’une forêt vers un autre, mais dans le cas d’une migration « nonmigratory », un des principes de base, vous devez appliquer est que les applications de la forêt initial doivent être actualisées, prise en charge et qui vient d’être installé. Données peuvent être migrées à partir d’instances d’application dans la forêt ancien lorsque cela est possible. Dans les cas dans lequel une application ne peut pas être « recréée » dans la forêt d’origine, vous devez envisager des approches, telles que la destruction creative ou l’isolement des applications héritées comme décrit dans la section suivante.  
  
### <a name="implementing-creative-destruction"></a>Implémentation de Destruction Creative  
Destruction Creative est un terme d’économie qui décrit le développement économique créé par la destruction d’une commande antérieure. Ces dernières années, le terme a été appliqué à l’informatique. Il fait généralement référence aux mécanismes par l’ancienne infrastructure est éliminé, ne pas par la mise à niveau, mais en le remplaçant par quelque chose de complètement nouveau. Le 2011 [Gartner Symposium ITXPO](http://www.gartner.com/technology/symposium/orlando/) pour les directeurs informatiques et les cadres informatiques présentées creative destruction comme l’un de ses principaux thèmes de réduction des coûts et augmente l’efficacité. Améliorations de sécurité sont possibles comme un prolongement naturel du processus.  

Par exemple, une organisation peut être composée de plusieurs divisions qui utilisent une autre application qui effectue des fonctions similaires, avec différents degrés de prise en charge modernity et fournisseur. Historiquement, IT est responsable de la maintenance d’application de chaque division séparément, et les efforts de consolidation consisterait en tente de déterminer quelle application a proposé une meilleure fonctionnalité et migrer ensuite des données dans qui application des autres.  
  
Dans creative destruction, plutôt que de gestion des applications obsolètes ou redondantes, vous allez implémenter des applications entièrement inédites pour remplacer l’ancien, de migrer des données dans les nouvelles applications et retirer les anciennes applications et les systèmes sur lesquels elles s’exécutent. Dans certains cas, vous pouvez implémenter creative destruction des applications héritées en déployant une nouvelle application dans votre propre infrastructure, mais dans la mesure du possible, vous devez envisager de portage de l’application à une solution basée sur le cloud à la place.  
  
En déployant des applications de cloud pour remplacer des applications héritées en interne, vous non seulement de réduire les coûts et les efforts de maintenance, mais vous réduisez la surface d’attaque de votre organisation en éliminant les systèmes hérités et les applications qui présentent des vulnérabilités des personnes malveillantes d’exploiter. Cette approche offre un moyen plus rapide pour une organisation obtenir la fonctionnalité souhaitée tout en éliminant simultanément cibles hérités dans l’infrastructure. Bien que le principe de destruction creative ne s’applique pas à toutes les ressources informatiques, il fournit une option viable souvent à l’élimination des systèmes hérités et des applications lors du déploiement d’applications robustes, sécurisées et basée sur le cloud simultanément.  
  
### <a name="isolating-legacy-systems-and-applications"></a>Isolation des Applications et systèmes existants  
Un prolongement naturel de la migration de vos systèmes et utilisateurs critiques vers un environnement sécurisé tel quel aux est que votre forêt hérité contient des informations moins intéressantes et les systèmes. Bien que les systèmes hérités et les applications qui restent dans l’environnement moins sécurisé peuvent présenter un risque élevé de compromission, elles représentent également une gravité réduite de compromission. En rapatriement et modernisation de vos ressources critiques, vous pouvez vous concentrer sur le déploiement d’une gestion efficace et la surveillance lors ne pas avoir à prendre en charge les protocoles et les paramètres hérités.  
  
Lorsque vous avez rapatriés vos données critiques dans une forêt vierge, vous pouvez évaluer les options pour isoler davantage les systèmes hérités et des applications dans votre forêt AD DS « main ». Bien que vous pouvez implémenter la destruction creative pour remplacer une application et les serveurs sur lesquels il s’exécute, dans d’autres cas, vous pouvez envisager une isolation supplémentaire des systèmes et des applications moins sécurisées. Par exemple, une application qui est utilisé par un petit nombre d’utilisateurs, mais qui nécessite des informations d’identification héritées tels que les hachages LAN Manager peuvent être migrées vers un domaine de petite taille crées pour prendre en charge les systèmes pour lesquels vous ne disposez d’aucune option de remplacement.  
  
En supprimant ces systèmes à partir de domaines où ils forcés d’implémentation de paramètres hérités, vous pouvez par la suite augmenter la sécurité des domaines en les configurant pour prendre en charge uniquement les applications et les systèmes d’exploitation actuels. Toutefois, il est préférable de mettre hors service des systèmes hérités et les applications autant que possible. Si la mise hors service n’est simplement pas possible pour un petit segment de votre population hérité, il séparant dans un domaine distinct (ou une forêt) vous permet d’effectuer des améliorations incrémentielles dans le reste de l’installation héritée.  
  
### <a name="simplifying-security-for-end-users"></a>Simplifiez-vous la sécurité pour les utilisateurs finaux  
Dans la plupart des organisations, les utilisateurs qui ont accès aux informations plus sensibles en raison de la nature de leurs rôles dans l’organisation souvent ont le moins de temps à allouer à des contrôles et des restrictions d’accès complexes d’apprentissage. Bien que vous devez avoir un programme de formation de sécurité complètes pour tous les utilisateurs dans votre organisation, vous devez également vous concentrer sur la création de la sécurité en tant que simples à utiliser que possible par l’implémentation de contrôles qui sont transparents et simplification des principes auquel les utilisateurs respecter.  
  
Par exemple, vous pouvez définir une stratégie dans laquelle cadres et autres adresses IP virtuelles doivent utiliser des stations de travail sécurisées pour accéder aux données sensibles et les systèmes, ce qui leur permet d’utiliser leurs autres appareils pour accéder aux données moins sensibles. Il s’agit d’un principe simple pour les utilisateurs à mémoriser, mais vous pouvez implémenter un nombre de contrôles de serveur principal pour aider à appliquer l’approche.  

Vous pouvez utiliser [Assurance du mécanisme d’authentification](https://technet.microsoft.com/library/dd391847(v=WS.10).aspx) pour autoriser les utilisateurs à accéder aux données sensibles uniquement si vous avez ouvert une session leurs systèmes sécurisés à l’aide de leurs cartes à puce, ils peuvent utiliser IPsec et les restrictions de contrôle des droits utilisateur le systèmes à partir de laquelle ils peuvent se connecter à des référentiels de données sensibles. Vous pouvez utiliser la [Microsoft Data Classification Toolkit](https://www.microsoft.com/download/details.aspx?id=27123) pour générer une infrastructure de classification de fichiers fiable et vous pouvez implémenter [contrôle d’accès dynamique](http://blogs.technet.com/b/windowsserver/archive/2012/05/22/introduction-to-windows-server-2012-dynamic-access-control.aspx) pour restreindre l’accès aux données basées sur caractéristiques d’une tentative d’accès, traduire des règles d’entreprise dans les contrôles techniques.  
  
Du point de vue de l’utilisateur, l’accès aux données sensibles à partir d’un système sécurisé « fonctionne, tout simplement » et essayer de le faire à partir d’un système non sécurisé « simplement ne pas. » Toutefois, du point de vue de la surveillance et la gestion de votre environnement, vous êtes vous aider à créer des modèles identifiables dans comment les utilisateurs aux données sensibles et les systèmes, ce qui facilite pour vous permettre de détecter les tentatives d’accès anormaux.  
  
Dans les environnements dans laquelle l’utilisateur a permis de résistance aux mots de passe longs et complexes des stratégies de mot de passe insuffisantes, en particulier pour les utilisateurs de l’adresse IP virtuelle, envisagez d’autres approches pour l’authentification, que ce soit par les cartes à puce (qui est fournie dans un nombre de facteurs de forme et avec des fonctionnalités supplémentaires pour renforcer l’authentification), contrôles biométriques comme les lecteurs de balayage de doigt ou des données même l’authentification sécurisée par approuvé puces de module de plateforme (TPM) dans les ordinateurs des utilisateurs. Bien que l’authentification multifacteur n’empêche pas le vol de ces informations d’identification si un ordinateur est déjà compromis, en permettant à vos utilisateurs de contrôles facile à utiliser l’authentification, vous pouvez affecter des mots de passe plus robustes pour les comptes d’utilisateurs pour lesquels traditionnel utilisateur les contrôles de nom et mot de passe sont peu maniable.
